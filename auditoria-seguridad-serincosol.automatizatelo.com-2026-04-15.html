Resumen Ejecutivo

Tu panel de gestión de fincas está desplegado en Vercel con Supabase como base de datos. No se encontraron vulnerabilidades críticas. La base de datos tiene RLS habilitado en todas las tablas con políticas correctas — el acceso anónimo está bloqueado. Quedan 3 hallazgos de severidad alta: el signup está abierto en Supabase (cualquiera puede crear cuenta), falta Content-Security-Policy, y no hay rate limiting. Son mejoras importantes antes de manejar datos sensibles de propietarios.

Analogía: Tu edificio digital

Imagina tu panel de fincas como un edificio de oficinas. La puerta principal (login) está cerrada con llave y vigilante — bien. Los archivadores internos (tablas de Supabase) tienen candado individual (RLS) — también bien. Sin embargo, la puerta de atrás (signup de Supabase) permite que cualquiera se fabrique su propia llave para entrar al edificio. No hay cámaras que limiten cuántas veces alguien puede intentar entrar (rate limiting). Y faltan carteles de seguridad en las ventanas (headers CSP) que evitarían que alguien pegue un cartel falso.

Vulnerabilidades de Severidad Alta

CORREGIDO   RLS en tablas de Supabase — VERIFICADO OK

La auditoría inicial detectó 8 tablas que respondían [] a consultas anónimas. Tras verificación manual en la base de datos, se confirmó que todas las 28 tablas tienen RLS habilitado (rowsecurity: true) y políticas configuradas exclusivamente para el rol authenticated. La respuesta [] es el comportamiento correcto de PostgREST cuando RLS bloquea el acceso al rol anon — no revela datos.

Estado: Falso positivo. La base de datos está correctamente protegida.

# Verificación: 28/28 tablas con rowsecurity = true
# Todas las políticas usan TO authenticated (no anon)
# Roles cubiertos: admin (full), gestor/empleado (por comunidad), owner (propios datos)

ALTA   Signup abierto en Supabase

La configuración de autenticación de Supabase tiene disable_signup: false. Esto significa que cualquier persona puede crear una cuenta en tu sistema. En una aplicación privada de gestión de fincas, solo los administradores deberían poder crear usuarios.

Impacto: Un atacante puede crear cuentas y, combinado con la falta de RLS, potencialmente acceder a todos los datos.

# Prueba de concepto
curl -X POST "https://dxpufxpzplkltljhevug.supabase.co/auth/v1/signup" \
  -H "apikey: [ANON_KEY]" -H "Content-Type: application/json" \
  -d '{"email":"test@example.com","password":"Password123!"}'
# Nota: el test con dominio .invalid fue rechazado por validacion de email,
# pero con un email real funcionaria

Fix: Deshabilitar signup público en Supabase Dashboard → Auth → Settings → Enable Sign Up → OFF. Crear usuarios via Admin API o invitaciones.

ALTA   Sin Content-Security-Policy (CSP)

El servidor no envía el header Content-Security-Policy. Este header es la defensa principal contra ataques XSS (Cross-Site Scripting), clickjacking avanzado e inyección de recursos maliciosos.

Impacto: Si un atacante encuentra un punto de inyección XSS, podría ejecutar JavaScript arbitrario, robar sesiones de usuario y acceder a datos de la aplicación.

# Header actual: NO presente
# Debería incluirse en la respuesta HTTP:
Content-Security-Policy: default-src 'self'; script-src 'self'; style-src 'self' 'unsafe-inline'

Fix: Añadir CSP en next.config.ts via headers:

// next.config.ts
async headers() {
  return [{
    source: '/(.*)',
    headers: [{
      key: 'Content-Security-Policy',
      value: "default-src 'self'; script-src 'self' 'unsafe-eval' 'unsafe-inline'; style-src 'self' 'unsafe-inline'; img-src 'self' data: https:; connect-src 'self' https://*.supabase.co"
    }]
  }]
}

ALTA   Sin Rate Limiting

Se enviaron 20 peticiones rápidas consecutivas y todas respondieron HTTP 200. No hay ningún mecanismo de limitación de velocidad, lo que permite ataques de fuerza bruta contra el login, enumeración de usuarios, y DDoS a nivel de aplicación.

# 20 peticiones rapidas — todas HTTP 200
for i in $(seq 1 20); do
  curl -s -o /dev/null -w "%{http_code} " "https://serincosol.automatizatelo.com/auth/login"
done
# Resultado: 200 200 200 200 200 200 200 200 200 200 200 200 200 200 200 200 200 200 200 200

Fix: Configurar rate limiting en Vercel (vercel.json) o implementar middleware de rate limiting en Next.js. Vercel ofrece WAF con rate limiting en planes Pro.

MEDIA   Auth Config públicamente accesible

El endpoint /auth/v1/settings de Supabase expone la configuración de autenticación: proveedores habilitados, estado de signup, autoconfirmación de email. Es comportamiento inherente a Supabase, pero proporciona información útil a un atacante.

curl "https://dxpufxpzplkltljhevug.supabase.co/auth/v1/settings" -H "apikey: [ANON_KEY]"
# Revela: disable_signup=false, mailer_autoconfirm=false, providers habilitados, etc.

Fix: No se puede deshabilitar. Mitigar asegurando que la configuración sea correcta (deshabilitar signup público).

Vulnerabilidades de Severidad Media

MEDIA   Sin X-Frame-Options ni frame-ancestors en CSP

No se envía el header X-Frame-Options. La aplicación podría ser embebida en un iframe por un sitio malicioso para ataques de clickjacking.

# Header no presente en respuesta

Fix: Añadir X-Frame-Options: DENY y frame-ancestors 'none' en CSP.

MEDIA   Sin Referrer-Policy

No se envía el header Referrer-Policy. Las URLs visitadas pueden filtrarse a sitios externos via el header Referer del navegador.

# Header no presente en respuesta

Fix: Añadir Referrer-Policy: strict-origin-when-cross-origin en headers de Next.js.

MEDIA   Sin Permissions-Policy

No se envía el header Permissions-Policy para restringir acceso a APIs del navegador como cámara, micrófono y geolocalización.

# Header no presente en respuesta

Fix: Añadir Permissions-Policy: camera=(), microphone=(), geolocation=().

MEDIA   Server Action endpoint descubierto

El endpoint de Server Actions de Next.js responde con "Server action not found", confirmando que el endpoint existe y acepta peticiones. Esto es una divulgación de información menor.

curl -X POST "https://serincosol.automatizatelo.com/" \
  -H "Next-Action: 0000000000" -H "Content-Type: text/plain" -d '[]'
# Respuesta: "Server action not found."

Fix: Comportamiento estándar de Next.js. Asegurar que las Server Actions validen autenticación internamente.

Plan de Acción

AHORA — Deshabilitar signup público

En Supabase Dashboard → Auth → Settings → desactivar "Enable Sign Up". Solo administradores deberían crear cuentas.

HOY — Añadir Content-Security-Policy

Configurar CSP en next.config.ts para prevenir XSS y otras inyecciones de contenido.

HOY — Añadir headers de seguridad faltantes

X-Frame-Options: DENY, Referrer-Policy: strict-origin-when-cross-origin, Permissions-Policy.

ESTA SEMANA — Implementar rate limiting

Usar middleware de Next.js o Vercel WAF (plan Pro) para limitar peticiones por IP, especialmente en login.

ANTES DE LANZAR — Configurar robots.txt

Añadir robots.txt para controlar qué indexan los buscadores. Considerar bloquear todo si es app privada.

ANTES DE LANZAR — Auditoría de RLS con datos reales

Una vez que haya datos y usuarios, repetir la auditoría para verificar que las políticas RLS funcionan correctamente y que un usuario no pueda ver datos de otro.

Implicaciones Legales (RGPD / LOPD-GDD)

Como plataforma que gestionará datos de comunidades de propietarios en España, está sujeta al Reglamento General de Protección de Datos (RGPD) y la Ley Orgánica 3/2018 (LOPD-GDD).

  • Art. 32 RGPD — Obligación de implementar medidas técnicas adecuadas (RLS, cifrado, control de acceso). Las tablas sin RLS incumplen este artículo.
  • Art. 33 RGPD — Obligación de notificar brechas a la AEPD en 72 horas.
  • Art. 83 RGPD — Sanciones de hasta 20M€ o 4% de facturación anual por incumplimiento de medidas de seguridad.
  • Datos de propietarios (nombres, DNI, direcciones, datos económicos) son datos personales protegidos.
  • Datos de morosos y deudas tienen protección reforzada como datos económicos.

Recomendación: El RLS ya está correctamente configurado. Resolver el signup abierto y añadir headers de seguridad ANTES de cargar datos reales de propietarios para evitar responsabilidad legal.
